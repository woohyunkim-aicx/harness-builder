# /harness-uninstall

Cleanly remove all files and configurations generated by harness-builder, preserving everything the user created or other plugins added.

**Input**: $ARGUMENTS
- `--dry-run` — Show what would be deleted without actually deleting
- `--force` — Skip confirmation prompt

---

## Pre-flight

1. Check if `.claude/harness-builder/harness-lock.json` exists
   - If not: "harness-builder로 생성된 설정이 없습니다." → exit
2. Read `harness-lock.json` to get the full list of managed files

---

## Step 1 — Inventory

Read `harness-lock.json.managedFiles` and categorize each entry:

### Category A: "full" managed files (safe to delete entirely)
These are files harness-builder created from scratch:
- `.claude/skills/{project-slug}-*/SKILL.md`
- `.claude/agents/{project-slug}-*.md`
- `.claude/scripts/{project-slug}-*.sh`
- `.claude/hooks/database-protection.json` (if managed)
- `.claude/docs/HARNESS-GUIDE.md`

### Category B: "merge" managed files (remove only our contributions)
These are shared files where we added entries alongside user/plugin entries:
- `.claude/settings.json` — Remove only deny patterns we added (NOT user's custom patterns)
- `.claude/settings.local.json` — Remove only allow patterns we added
- `.claude/hooks/hooks.json` — Remove only entries with `[harness-builder]` in description

### Category C: "sections" managed files (remove only our markers)
- `CLAUDE.md` — Remove content between `<!-- harness-builder:start:* -->` and `<!-- harness-builder:end:* -->` markers, including the markers themselves

### Category D: harness-builder internal files (always delete)
- `.claude/harness-builder/` directory entirely (analysis.json, interview.json, harness-lock.json, backups/)

---

```

  - .claude/scripts/voc-tool-tsc-check.sh
  - .claude/docs/HARNESS-GUIDE.md

✏️  부분 수정 (Category B — 우리가 추가한 항목만 제거):
  - .claude/settings.json → {N}개 deny 패턴 제거
  - .claude/hooks/hooks.json → [harness-builder] 태그된 {N}개 훅 제거

📝 섹션 제거 (Category C — 마커 사이 내용만):
  - CLAUDE.md → {N}개 관리 섹션 제거 (agent-routing, model-policy, ...)
```

If `--dry-run`: print and exit.

## Step 3 — Confirmation

If NOT `--force`:

Ask user via AskUserQuestion:
- Question: "위 항목들을 삭제할까요? 백업은 .claude/harness-builder/backups/에 보존됩니다."
- Header: "삭제 확인"
- Options:
  - "삭제 진행" — Proceed with deletion
  - "백업 먼저 생성 후 삭제" — Create fresh backup, then delete
  - "취소" — Abort

If "취소": exit.
If "백업 먼저": run `bash ${CLAUDE_PLUGIN_ROOT}/scripts/backup-existing.sh` first.

---

## Step 4 — Execute Deletion

Process in reverse dependency order (delete dependents first):

### 4.1 Category A: Delete full files
```bash
# For each file in managedFiles where managed == "full"
rm -f ".claude/agents/{project-slug}-*.md"
rm -rf ".claude/skills/{project-slug}-*/"
rm -f ".claude/scripts/{project-slug}-*.sh"
rm -f ".claude/docs/HARNESS-GUIDE.md"
rm -f ".claude/hooks/database-protection.json"  # if managed
```

Verify each file exists before deleting. Log each deletion.

### 4.2 Category B: Clean shared JSON files

**settings.json** — Remove only harness-builder deny patterns:
1. Read current settings.json
2. Read harness-lock.json to find which deny patterns were added by us
   - Compare current deny array with the template deny patterns from `${CLAUDE_PLUGIN_ROOT}/templates/settings/settings.json.tmpl`
   - Remove ONLY exact matches with template patterns
3. If deny array becomes empty, remove the `permissions.deny` key
4. If permissions object becomes empty, remove it
5. If entire file becomes `{}`, delete the file
6. Otherwise, write back the cleaned file

**settings.local.json** — Same approach for allow patterns.

**hooks.json** — Remove `[harness-builder]` tagged entries:
1. Read current hooks.json
2. For each hook point (PreToolUse, PostToolUse, Stop, etc.):
   - Filter out entries where `description` starts with `[harness-builder]`
   - Keep all other entries intact
3. If a hook point array becomes empty, remove that key
4. If hooks object becomes empty, delete the file
5. Otherwise, write back

### 4.3 Category C: Clean CLAUDE.md sections

1. Read CLAUDE.md
2. For each managed section marker pair:
   - Find `<!-- harness-builder:start:{name} -->`
   - Find `<!-- harness-builder:end:{name} -->`
   - Remove everything from start marker line to end marker line (inclusive)
3. Clean up resulting double blank lines (replace 3+ blank lines with 2)
4. If CLAUDE.md becomes effectively empty (only whitespace/blank lines), delete it
5. Otherwise, write back

Also remove the coexistence note comment if present:
```
<!-- Note: Sections between harness-builder markers are auto-managed. ... -->
```

### 4.4 Category D: Delete internal data

```bash
# Move backups to a temp location BEFORE deleting harness-builder dir
BACKUP_PRESERVE="/tmp/harness-builder-backups-$(date +%s)"
if [ -d ".claude/harness-builder/backups" ]; then
  cp -r .claude/harness-builder/backups "$BACKUP_PRESERVE"
fi

rm -rf .claude/harness-builder/
```

### 4.5 Clean up empty directories

```bash
# Remove empty directories left behind
find .claude/skills/ -type d -empty -delete 2>/dev/null
find .claude/agents/ -type d -empty -delete 2>/dev/null
find .claude/scripts/ -type d -empty -delete 2>/dev/null
find .claude/docs/ -type d -empty -delete 2>/dev/null
find .claude/hooks/ -type d -empty -delete 2>/dev/null

# If .claude/ itself is now empty (only had harness-builder content), remove it
if [ -z "$(ls -A .claude/ 2>/dev/null)" ]; then
  rmdir .claude/
fi
```

## Step 5 — Verification

After deletion, verify:

1. [ ] No harness-builder managed files remain (glob for `{project-slug}-*` patterns)
2. [ ] No orphaned section markers in CLAUDE.md (grep for `harness-builder:`)
3. [ ] No `[harness-builder]` tagged hooks remain in hooks.json
4. [ ] settings.json still valid JSON (if file exists)
5. [ ] hooks.json still valid JSON (if file exists)
6. [ ] CLAUDE.md still readable (if file exists)
7. [ ] No other plugin files were touched

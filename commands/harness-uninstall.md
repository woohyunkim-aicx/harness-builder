# /harness-uninstall

Cleanly remove all files and configurations generated by harness-builder, preserving everything the user created or other plugins added.

**Input**: $ARGUMENTS
- `--dry-run` — Show what would be deleted without actually deleting
- `--force` — Skip confirmation prompt

---

## Pre-flight

1. Check if `.claude/harness-builder/harness-lock.json` exists
   - If not: "harness-builder로 생성된 설정이 없습니다." → exit

2. Read the raw content of `harness-lock.json` and validate it is well-formed JSON before doing anything else.
   - If the file is empty or not valid JSON: "harness-lock.json이 손상되어 있습니다. 수동으로 확인해 주세요." → exit
   - Do NOT proceed with any deletion if the lock file cannot be parsed.

3. Parse `harness-lock.json` and extract:
   - `managedFiles` — object whose keys are relative file paths and values describe how the file was managed (`"full"`, `"merge"`, `"sections"`)
   - `addedDenyPatterns` — array of exact strings that were inserted into `settings.json`
   - `addedHookEntries` — array of hook entry objects that were inserted into `hooks.json`
   - `claudeMdSections` — array of section name strings managed in `CLAUDE.md`
   - `projectSlug` — the project slug used during install

---

## Step 1 — Inventory

Using **only** the explicit data from `harness-lock.json`, categorize each entry:

### Category A: "full" managed files (safe to delete entirely)
Files whose `managedFiles[path]` value is `"full"`. These were created from scratch by harness-builder. Examples:
- `.claude/skills/{projectSlug}-*/SKILL.md` entries
- `.claude/agents/{projectSlug}-*.md` entries
- `.claude/scripts/{projectSlug}-*.sh` entries
- `.claude/docs/HARNESS-GUIDE.md`
- `.claude/hooks/database-protection.json` (if present in managedFiles)

**Only delete the exact paths listed as keys in `managedFiles`. Never use glob patterns for deletion.**

### Category B: "merge" managed files (remove only tracked contributions)
Files whose `managedFiles[path]` value is `"merge"`. These are shared files where harness-builder added entries alongside other content:
- `.claude/settings.json` — Remove only the strings listed in `addedDenyPatterns`
- `.claude/settings.local.json` — Remove only the strings listed in `addedAllowPatterns` (if tracked)
- `.claude/hooks/hooks.json` — Remove only the exact entry objects listed in `addedHookEntries`

### Category C: "sections" managed files (remove only our markers and their content)
Files whose `managedFiles[path]` value is `"sections"`:
- `CLAUDE.md` — Remove content between `<!-- harness-builder:start:{name} -->` and `<!-- harness-builder:end:{name} -->` marker pairs, for each name listed in `claudeMdSections`

### Category D: harness-builder internal directory (always delete last)
- `.claude/harness-builder/` directory entirely

## Step 4 — Execute Deletion

Process in this exact order. If any individual step fails, log the failure and continue with remaining steps — do not abort entirely. Collect all failures and report them in the completion summary.

### 4.0 Backup (if requested or --backup flag present)

Create backup at `.claude/harness-builder-uninstall-backup-{timestamp}/` — this location is **outside** the `harness-builder/` directory so it survives the deletion of Category D.

```bash
TIMESTAMP=$(date +%Y%m%d-%H%M%S)
BACKUP_DIR=".claude/harness-builder-uninstall-backup-${TIMESTAMP}"
mkdir -p "$BACKUP_DIR"

# Copy every file that will be modified or deleted
# Use explicit paths from managedFiles, not globs
cp .claude/harness-builder/harness-lock.json "$BACKUP_DIR/"
# Copy each Category A file if it exists
# Copy Category B files (settings.json, hooks.json) if they exist
# Copy Category C files (CLAUDE.md) if they exist
```

Do NOT use `/tmp/` — backups must stay within the project.

### 4.1 Category A: Delete exact files from managedFiles

Iterate over **each key** in `managedFiles` where the value is `"full"`. Delete that exact path. Do not expand globs or infer additional paths.

```bash
# Pseudocode — execute for each "full" entry in managedFiles
for path in managedFiles.fullEntries:
    if file_exists(path):
        rm -f "$path"
        log "삭제됨: $path"
    else:
        log "건너뜀 (존재하지 않음): $path"
```

After deleting all "full" files, check if any **parent directories that were themselves listed as managed** (i.e., are also keys in `managedFiles` with value `"full"`) are now empty. If so, remove those directories. Do NOT run `find -empty -delete` across any directory tree — only remove the specific directories whose paths appear in `managedFiles`.

```bash
# Only remove directories explicitly listed in managedFiles as "full"
for dir in managedFiles.fullDirectoryEntries:
    if dir_exists(dir) and dir_is_empty(dir):
        rmdir "$dir"
        log "빈 디렉토리 삭제됨: $dir"
    elif dir_exists(dir):
        log "보존됨 (비어있지 않음): $dir"
```

### 4.2 Category B: Clean shared JSON files

**settings.json** — Remove only deny patterns tracked in `addedDenyPatterns`:

1. Read current `.claude/settings.json`
2. For each string in `harness-lock.json → addedDenyPatterns`:
   - Remove that exact string from the `permissions.deny` array if present
   - If the string is NOT found, log a warning but continue
3. If `permissions.deny` becomes empty, remove the `deny` key
4. If `permissions` object becomes empty, remove it
5. If the entire file becomes `{}`, delete the file
6. Otherwise, write back the cleaned JSON (pretty-printed, 2-space indent)
7. If the file does not exist, skip with a note

**settings.local.json** — Same approach using `addedAllowPatterns` for the allow array.

**hooks.json** — Remove only hook entries tracked in `addedHookEntries`:

1. Read current `.claude/hooks/hooks.json`
2. For each entry object in `harness-lock.json → addedHookEntries`:
   - Identify its hook point (e.g., `PreToolUse`, `PostToolUse`, `Stop`) and match by the entry's unique identifier (use `description` field as the key — it must be unique)
   - Remove only that exact entry from the array
   - If the entry is NOT found in hooks.json, log a warning but continue
3. If a hook point array becomes empty, remove that key from the hooks object
4. If the hooks object becomes empty, delete the file
5. Otherwise, write back the cleaned JSON

Do NOT filter by `[harness-builder]` tag pattern alone — use the explicit entries from `addedHookEntries` as the source of truth.

### 4.3 Category C: Clean CLAUDE.md sections

For each section name in `harness-lock.json → claudeMdSections`:

1. Read CLAUDE.md
2. Search for the marker pair:
   - Start: `<!-- harness-builder:start:{name} -->`
   - End: `<!-- harness-builder:end:{name} -->`
3. **Validate before removing**: confirm both markers exist AND each appears on its own line (not embedded in other content). If either marker is missing or appears malformed, skip this section with a warning — do NOT attempt partial removal.
4. Remove everything from the start marker line through the end marker line (inclusive)
5. Clean up resulting consecutive blank lines (replace 3+ consecutive blank lines with 2)

After processing all sections:
- If CLAUDE.md becomes effectively empty (only whitespace/blank lines remain), delete it
- Otherwise, write back

Also remove the coexistence note comment if present as a standalone line:
```
<!-- Note: Sections between harness-builder markers are auto-managed. ... -->
```
Only remove it if it exists as a distinct line — do not regex-delete across content.

### 4.4 Category D: Delete internal directory

Delete `.claude/harness-builder/` **after** all other categories are complete (the lock file is needed until this point).

```bash
rm -rf .claude/harness-builder/
log "삭제됨: .claude/harness-builder/"
```

### 4.5 .claude/ directory cleanup

After all deletions, check if `.claude/` itself is now completely empty:

```bash
if [ -d ".claude" ] && [ -z "$(ls -A .claude/ 2>/dev/null)" ]; then
  rmdir .claude/
  log ".claude/ 디렉토리가 비어 있어 삭제됨"
else
  log ".claude/ 디렉토리 보존됨 (다른 내용 존재)"
fi
```

Do NOT use `find -empty -delete` on `.claude/` or any subdirectory. Only check and conditionally remove `.claude/` itself.

## Reference: harness-lock.json Schema (v2)

The `config-generator` must write a `harness-lock.json` conforming to this schema to enable safe uninstall. All fields used during uninstall are marked **required for uninstall**.

```jsonc
{
  "schemaVersion": "2",           // required — bump when schema changes
  "projectSlug": "voc-tool",      // required for uninstall

  // Keys are relative paths from project root.
  // Values: "full" | "merge" | "sections"
  "managedFiles": {
    ".claude/agents/voc-tool-backend-developer.md": "full",
    ".claude/skills/voc-tool-api-patterns/SKILL.md": "full",
    ".claude/skills/voc-tool-api-patterns": "full",  // parent dir entry
    ".claude/scripts/voc-tool-tsc-check.sh": "full",
    ".claude/docs/HARNESS-GUIDE.md": "full",
    ".claude/settings.json": "merge",
    ".claude/hooks/hooks.json": "merge",
    "CLAUDE.md": "sections"
  },

  // REQUIRED FOR UNINSTALL — exact strings added to settings.json deny array.
  // Must be recorded at install time. Never reconstruct from templates.
  "addedDenyPatterns": [
    "Bash(rm -rf /*)",
    "Bash(DROP TABLE*)"
  ],

  // Optional — exact strings added to settings.local.json allow array
  "addedAllowPatterns": [],

  // REQUIRED FOR UNINSTALL — exact hook entry objects inserted into hooks.json.
  // Each object must include the hook point as a top-level key so uninstall
  // knows which array to remove the entry from.
  "addedHookEntries": [
    {
      "hookPoint": "PreToolUse",   // which hooks.json array this belongs to
      "matcher": "Bash",
      "description": "[harness-builder] Block dangerous Bash commands",
      "command": "bash .claude/scripts/voc-tool-tsc-check.sh"
    }
  ],

  // REQUIRED FOR UNINSTALL — section names managed in CLAUDE.md.
  // Must match the {name} in <!-- harness-builder:start:{name} --> markers exactly.
  "claudeMdSections": [
    "agent-routing",
    "model-policy"
  ],

  // Files that existed before install and were intentionally skipped
  "skippedFiles": [],

  // ISO 8601 timestamps
  "installedAt": "2026-04-08T09:00:00Z",
  "updatedAt": "2026-04-08T09:00:00Z"
}
```

**Key invariant**: every string in `addedDenyPatterns`, every object in `addedHookEntries`, and every name in `claudeMdSections` must be recorded at install time from the actual values written — not inferred from templates or reconstructed at uninstall time.
